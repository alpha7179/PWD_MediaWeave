## 1. 프로젝트 개요 (Introduction)

### 1.1 서비스 비전 (Vision Statement)

> "우리의 이야기를 한곳에, 디미를 잇다."

아주대학교 디지털미디어학과 재학생의 불투명한 진로 설계와 파편화된 성과 관리 문제 해결을 위한 학과 특화형 커리어 플랫폼. 학생 개개인의 자기주도적 커리큘럼 설계(Weave), 프로젝트 아카이빙, 산업 현장 수준의 포트폴리오 완성 지원. 학계 교육 성과와 산업계 인재 수요 연결.

### 1.2 개발 배경 및 필요성

본 프로젝트의 4가지 핵심 문제 해결:

1. **정보 비대칭 해소**: 복잡한 선수과목 관계와 5개 트랙(DE, CD, GI, MD, VC) 정보의 시각화를 통한 정보 격차 감소
2. **커리큘럼 가시화**: 텍스트 중심 요람 대신 실제 결과물 연계 로드맵 제공
3. **학술 자산 아카이빙**: 학생 우수 프로젝트의 중앙 저장소 축적을 통한 학과 자산화
4. **효율적 팀 빌딩**: 역량 기반 팀 매칭 시스템 구축

## 2. 사용자 정의 및 페르소나 (User Definition)

학업 단계별 명확한 니즈를 가진 능동적 주체로서의 사용자 정의.

### 👥 Persona A: 김후배 (2학년, 정보 탐색자)

- **상황**: 전공 진입 초기 단계, 트랙 선택 및 과목 간 연계성 파악의 어려움
- **Needs**: GI 트랙 진입을 위한 3학년 필수 과목 정보, 선배들의 수강 경로 참고
- **목표**: 실패 없는 수강 신청과 효율적인 졸업 설계

### 👥 Persona B: 이선배 (4학년, 콘텐츠 생산자)

- **상황**: 졸업 작품 및 취업 준비 단계, 프로젝트 정리 및 협업자 필요
- **Needs**: 단일 URL 포트폴리오, 졸업 작품 웹 개발 협업자 매칭
- **목표**: 역량 증명(Verification) 및 질 높은 협업(Collaboration)

## 3. 기능 요구사항 (Functional Requirements)

### 3.1 인터랙티브 트랙 로드맵 (Track Roadmap)

**목적**: 시각적 정보 탐색을 통한 트랙별 추천 경로 제공

#### 입력 (Input)
- 사용자 선택 트랙 (DE, CD, GI, MD, VC 중 1개)
- 과목 노드 클릭 이벤트
- 필터 조건 (학년, 학기, 이수 여부)

#### 처리 (Processing)
- 선택 트랙에 따른 과목 그래프 필터링
- 과목 간 선수과목 관계(Dependency) 계산
- 노드 레이아웃 자동 배치 (React Flow 라이브러리)
- 과목별 우수 프로젝트 조회 (최대 3개)

#### 출력 (Output)
- 트랙별 과목 노드 그래프 (유기적 연결선 포함)
- 과목 상세 모달 (과목명, 학점, 선수과목, 개요, 우수 프로젝트 썸네일)
- 선수과목 미이수 시 시각적 경고 표시 (노드 색상 변경)

#### 사용자 시나리오
1. 사용자가 GI 트랙 탭 클릭
2. 시스템이 GI 트랙 관련 과목 노드 그래프 렌더링 (0.5초 이내)
3. 사용자가 "게임 엔진 프로그래밍" 노드 클릭
4. 시스템이 과목 상세 모달 표시 (과목 개요 + 우수 프로젝트 3개)
5. 사용자가 선수과목 "자료구조" 미이수 상태인 경우, 노드에 빨간색 테두리 표시

#### 예외 상황 및 처리
| 예외 상황 | 처리 방법 | 사용자 피드백 |
|----------|----------|--------------|
| 트랙 데이터 로딩 실패 | 3회 재시도 후 에러 페이지 표시 | "트랙 정보를 불러올 수 없습니다. 새로고침 해주세요." |
| 과목 상세 정보 없음 | 기본 정보만 표시 (과목명, 학점) | "상세 정보가 준비 중입니다." |
| 우수 프로젝트 없음 | 빈 상태 메시지 표시 | "아직 등록된 프로젝트가 없습니다." |
| 네트워크 타임아웃 (5초 초과) | 로딩 스피너 유지 후 에러 토스트 | "네트워크 연결을 확인해주세요." |

#### 검증 기준
- [ ] 5개 트랙 모두 정상 렌더링 (각 트랙당 15개 이상 과목 노드)
- [ ] 노드 클릭 시 모달 표시 시간 < 200ms
- [ ] 선수과목 관계 정확도 100% (학과 요람 기준)
- [ ] 그래프 레이아웃 자동 배치 시 노드 겹침 0건
- [ ] 모바일 환경에서 터치 제스처 (핀치 줌, 드래그) 정상 동작

### 3.2 나만의 커리큘럼 플래너 (My Curriculum Planner)

**목적**: 8학기 수강 계획 수립 및 시뮬레이션

#### 입력 (Input)
- AIMS 성적표 CSV 파일 (선택)
- 과목 드래그 앤 드롭 이벤트
- 학기별 과목 배치 (1학년 1학기 ~ 4학년 2학기)
- 과목 상태 변경 (수강 예정 / 수강 중 / 이수 완료)

#### 처리 (Processing)
- CSV 파싱 및 기이수 과목 자동 매칭
- 학기별 학점 합계 실시간 계산
- 선수과목 유효성 검사 (프론트엔드 + 백엔드)
- 졸업 요건 충족 여부 계산 (전공 필수, 전공 선택, 교양, 총 학점)
- 커리큘럼 데이터 자동 저장 (3초마다)

#### 출력 (Output)
- 8학기 그리드 레이아웃 (각 학기당 최대 7과목)
- 학기별 학점 합계 (실시간 업데이트)
- 졸업 요건 진행률 (전공 필수 12/15, 전공 선택 24/30 등)
- 선수과목 미이수 경고 배지
- 저장 상태 표시 (저장 중 / 저장 완료)

#### 사용자 시나리오
1. 사용자가 AIMS 성적표 CSV 업로드
2. 시스템이 CSV 파싱 후 기이수 과목 자동 배치 (2학년 1학기까지 완료 상태)
3. 사용자가 "게임 엔진 프로그래밍"을 3학년 1학기로 드래그
4. 시스템이 선수과목 "자료구조" 미이수 감지 후 경고 표시
5. 사용자가 "자료구조"를 2학년 2학기로 먼저 배치
6. 시스템이 유효성 검사 통과 후 자동 저장

#### 예외 상황 및 처리
| 예외 상황 | 처리 방법 | 사용자 피드백 |
|----------|----------|--------------|
| CSV 형식 오류 | 파싱 실패 시 에러 모달 표시 | "올바른 AIMS 성적표 파일을 업로드해주세요." |
| 학기당 학점 초과 (21학점 이상) | 드롭 불가 처리 | "학기당 최대 21학점까지 가능합니다." |
| 선수과목 미이수 | 경고 배지 표시 (드롭은 허용) | "선수과목을 먼저 이수해야 합니다." |
| 자동 저장 실패 | 3회 재시도 후 수동 저장 버튼 활성화 | "자동 저장 실패. 수동 저장 버튼을 눌러주세요." |
| 중복 과목 배치 | 드롭 불가 처리 | "이미 배치된 과목입니다." |

#### 검증 기준
- [ ] CSV 업로드 후 기이수 과목 자동 매칭 정확도 100%
- [ ] 드래그 앤 드롭 응답 시간 < 100ms
- [ ] 학기별 학점 합계 실시간 계산 (입력 후 즉시 반영)
- [ ] 선수과목 유효성 검사 정확도 100% (프론트엔드 + 백엔드)
- [ ] 자동 저장 성공률 > 99% (3초 간격)
- [ ] 졸업 요건 계산 정확도 100% (학과 규정 기준)

### 3.3 프로젝트 갤러리 & 아카이브 (Project Archive)

**목적**: 학과 학술 자산의 디지털 전시 및 아카이빙

#### 입력 (Input)
- 검색 키워드 (프로젝트명, 작성자명)
- 다중 필터 (기술 스택, 학년, 트랙, 과목, 연도)
- 정렬 기준 (최신순, 인기순, 조회수순)
- 무한 스크롤 페이지네이션 트리거

#### 처리 (Processing)
- 검색어 기반 전문 검색 (Full-text Search)
- 다중 필터 조합 쿼리 실행
- 썸네일 이미지 CDN 캐싱 (CloudFront)
- 조회수 증가 (중복 방지: IP + 세션 기반)
- 좋아요 토글 처리 (사용자당 1회)

#### 출력 (Output)
- 카드형 그리드 레이아웃 (데스크톱 4열, 태블릿 3열, 모바일 2열)
- 프로젝트 카드 (썸네일, 제목, 작성자, 기술 스택 태그, 좋아요 수, 조회수)
- 필터 적용 결과 개수 (예: "총 47개 프로젝트")
- 빈 상태 메시지 (검색 결과 없음)

#### 사용자 시나리오
1. 사용자가 "Unity" 기술 스택 필터 선택
2. 시스템이 Unity 태그 포함 프로젝트 조회 (0.3초 이내)
3. 사용자가 추가로 "GI 트랙" 필터 선택
4. 시스템이 Unity + GI 조합 프로젝트 재조회 (총 12개)
5. 사용자가 프로젝트 카드 클릭
6. 시스템이 조회수 +1 증가 후 상세 페이지 이동

#### 예외 상황 및 처리
| 예외 상황 | 처리 방법 | 사용자 피드백 |
|----------|----------|--------------|
| 검색 결과 없음 | 빈 상태 일러스트 + 필터 초기화 버튼 | "검색 결과가 없습니다. 필터를 조정해보세요." |
| 썸네일 로딩 실패 | 기본 플레이스홀더 이미지 표시 | (시각적 대체, 메시지 없음) |
| CDN 장애 | S3 원본 이미지로 폴백 | (사용자 인지 불가, 로딩 시간 증가) |
| 좋아요 API 실패 | 낙관적 업데이트 후 롤백 | "일시적 오류가 발생했습니다." (토스트) |
| 무한 스크롤 로딩 실패 | 재시도 버튼 표시 | "더 보기 실패. 다시 시도해주세요." |

#### 검증 기준
- [ ] 검색 응답 시간 < 300ms (인덱스 적용)
- [ ] 썸네일 이미지 로딩 시간 < 500ms (CDN 캐싱)
- [ ] 다중 필터 조합 정확도 100%
- [ ] 조회수 중복 카운트 방지율 100% (동일 IP + 세션)
- [ ] 무한 스크롤 페이지당 로딩 개수 20개
- [ ] 모바일 환경 카드 레이아웃 반응형 정상 동작

### 3.4 역량 기반 팀 빌딩 (Team Building)

**목적**: 프로젝트 협업자 매칭 및 팀 구성

#### 입력 (Input)
- 팀 모집 게시글 (제목, 설명, 기간, 모집 인원)
- 필요 기술 스택 태그 (최소 1개, 최대 10개)
- 필요 역할 (기획자, 디자이너, 프론트엔드, 백엔드 등)
- 지원 신청 (지원 메시지, 포트폴리오 링크)

#### 처리 (Processing)
- 작성자 프로필 자동 연동 (이수 과목, 과거 프로젝트, 기술 스택)
- 기술 스택 기반 추천 알고리즘 (매칭 점수 계산)
- 모집 상태 자동 업데이트 (모집 인원 충족 시 마감)
- 지원자 알림 발송 (이메일 + 인앱 알림)
- 모집 마감일 자동 처리 (D-day 계산)

#### 출력 (Output)
- 팀 모집 게시글 목록 (카드형 레이아웃)
- 모집 상태 배지 (모집중: 초록색, 마감: 회색)
- 작성자 프로필 미리보기 (이름, 학년, 대표 프로젝트 1개)
- 매칭 점수 표시 (본인 기술 스택과 일치율 %)
- 지원자 목록 (작성자만 조회 가능)

#### 사용자 시나리오
1. 사용자가 "졸업 작품 웹 개발자 구함" 게시글 작성
2. 시스템이 작성자 프로필 자동 연동 (Unity, C# 기술 보유)
3. 필요 기술 스택 "React, TypeScript" 태그 추가
4. 시스템이 해당 기술 보유 사용자에게 추천 알림 발송
5. 지원자가 지원 메시지 + 포트폴리오 링크 제출
6. 작성자가 지원자 프로필 확인 후 수락/거절

#### 예외 상황 및 처리
| 예외 상황 | 처리 방법 | 사용자 피드백 |
|----------|----------|--------------|
| 기술 스택 태그 없음 | 게시글 작성 불가 | "최소 1개 이상의 기술 스택을 선택해주세요." |
| 모집 인원 0명 | 게시글 작성 불가 | "모집 인원은 1명 이상이어야 합니다." |
| 중복 지원 | 지원 불가 처리 | "이미 지원한 게시글입니다." |
| 마감된 게시글 지원 | 지원 불가 처리 | "마감된 모집 공고입니다." |
| 본인 게시글 지원 | 지원 불가 처리 | "본인이 작성한 게시글에는 지원할 수 없습니다." |

#### 검증 기준
- [ ] 작성자 프로필 자동 연동 정확도 100%
- [ ] 매칭 점수 계산 정확도 (기술 스택 일치율 기반)
- [ ] 모집 상태 자동 업데이트 (인원 충족 시 즉시 마감)
- [ ] 지원 알림 발송 성공률 > 99%
- [ ] 중복 지원 방지율 100%
- [ ] 마감일 자동 처리 정확도 100% (매일 00:00 배치 작업)

### 3.5 통합 포트폴리오 (My Profile)

**목적**: 단일 URL 기반 취업용 포트폴리오 제공

#### 입력 (Input)
- 프로필 정보 (이름, 학번, 학년, 트랙, 연락처)
- 자기소개 (Markdown 형식 지원)
- 외부 링크 (GitHub, LinkedIn, 개인 블로그)
- 대표 프로젝트 선택 (최대 6개)
- 공개 범위 설정 (전체 공개 / 링크만 공개 / 비공개)

#### 처리 (Processing)
- Markdown 파싱 및 HTML 변환
- 프로젝트 자동 정렬 (최신순 / 인기순)
- 이수 과목 자동 집계 (커리큘럼 플래너 연동)
- Open Graph 메타 태그 생성 (공유 시 미리보기)
- 프로필 조회수 증가 (중복 방지: IP + 세션 기반)

#### 출력 (Output)
- 좌측 사이드바 (프로필 사진, 이름, 학년, 트랙, 연락처, 외부 링크)
- 우측 메인 영역 (탭: Projects, Posts, Curriculums)
- Projects 탭: 대표 프로젝트 그리드 (썸네일, 제목, 기술 스택)
- Posts 탭: 작성한 팀 빌딩 게시글 목록
- Curriculums 탭: 8학기 커리큘럼 시각화
- 공유 버튼 (URL 복사, SNS 공유)

#### 사용자 시나리오
1. 사용자가 프로필 편집 페이지 진입
2. Markdown 에디터로 자기소개 작성 (실시간 미리보기)
3. 대표 프로젝트 6개 선택 (드래그로 순서 변경)

5. 공개 범위 "전체 공개" 선택 후 저장
6. 시스템이 고유 URL 생성 (예: /profile/2021123456)
7. 사용자가 URL 복사 후 취업 지원서에 첨부

### 5.2 데이터베이스 설계 원칙

#### 정규화
- 제3정규형 (3NF) 준수
- 중복 데이터 최소화
- 참조 무결성 보장 (Foreign Key Constraints)

#### 인덱싱 전략
- Primary Key: 자동 인덱스
- Foreign Key: 인덱스 추가
- 검색 빈도 높은 컬럼: 단일 인덱스
- 복합 검색 조건: 복합 인덱스
- 전문 검색: GIN 인덱스 (PostgreSQL)

#### 관계 무결성
- ON DELETE CASCADE: 부모 삭제 시 자식 자동 삭제 (예: User 삭제 시 Projects 삭제)
- ON DELETE SET NULL: 부모 삭제 시 자식 NULL 처리 (예: Track 삭제 시 User.track_id NULL)
- ON DELETE RESTRICT: 자식 존재 시 부모 삭제 불가 (예: Course 삭제 시 Curriculum 존재하면 불가)

### 5.3 API 설계 원칙

#### RESTful 규칙
- 리소스 중심 URL 설계: `/api/v1/users`, `/api/v1/projects`
- HTTP 메서드 의미 준수: GET (조회), POST (생성), PUT (전체 수정), PATCH (부분 수정), DELETE (삭제)
- 상태 코드 정확한 사용: 200 (성공), 201 (생성), 400 (잘못된 요청), 401 (인증 실패), 403 (권한 없음), 404 (없음), 500 (서버 오류)

#### 응답 형식 표준화
```json
{
  "success": true,
  "data": { ... },
  "message": "요청이 성공적으로 처리되었습니다.",
  "timestamp": "2025-12-08T10:30:00Z"
}
```

#### 에러 응답 형식
```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "입력 값이 올바르지 않습니다.",
    "details": [
      { "field": "email", "message": "이메일 형식이 아닙니다." }
    ]
  },
  "timestamp": "2025-12-08T10:30:00Z"
}
```

### 5.4 보안 아키텍처

#### 인증 플로우
1. 사용자 로그인 (이메일 + 비밀번호)
2. 서버가 Bcrypt로 비밀번호 검증
3. JWT Access Token (1시간) + Refresh Token (7일) 발급
4. 클라이언트가 Access Token을 Authorization 헤더에 포함
5. Access Token 만료 시 Refresh Token으로 재발급

#### 권한 체크
- Middleware: `authMiddleware` (JWT 검증)
- Middleware: `roleMiddleware` (역할 검증)
- 소유권 검증: 본인 데이터만 수정 가능

### 5.5 배포 아키텍처

   ↓
[Route 53 DNS]
   ↓
[ALB Load Balancer]
   ↓
[EC2 Instance 1] [EC2 Instance 2]
   ↓

#### 기술 지표
- 서비스 가용성 > 99.9%
- API 응답 시간 P95 < 200ms
- 페이지 로딩 시간 < 1.5초
- 에러율 < 0.1%

### 7.2 정성적 효과

#### 학생 (Users)
- 시행착오 없는 효율적인 커리어 설계
- 고품질 포트폴리오 확보 (단일 URL)
- 역량 기반 팀 매칭을 통한 협업 기회 증가
- 선배들의 수강 경로 참고를 통한 정보 격차 해소

#### 학과 (Department)
- 커리큘럼 정보의 투명성 제고
- 재학생 전공 역량 상향 평준화
- 학과 우수 프로젝트 아카이빙을 통한 자산화
- 학과 홍보 자료로 활용 가능

#### 산업 (Industry)
- 검증된 이수 내역과 프로젝트 경험 보유 인재 채용
- 기업 맞춤형 기술 스택 보유 인재 탐색 용이
- 채용 프로세스 효율화 (포트폴리오 사전 검토)
